# NPCine Project Architecture

## Overview
NPCine is a microservices-based cinema booking platform. All client traffic should enter through the API Gateway and then be routed to domain-specific services.

Core domains:
- Identity and authentication (`user-service`)
- Seat inventory and booking lifecycle (`booking-service`)
- Payment orchestration and events (`payment-service`)
- Asynchronous user notifications (`notification_service`)

## Service Topology

### Entry Point
- `api-gateway` (`:8081`)
  - Routes public/authenticated requests to internal services.
  - Validates JWT for protected paths.

### Domain Services
- `user-service` (`:8080`, MongoDB)
  - Auth endpoints: `/auth/register`, `/auth/login`
  - Issues JWT used by downstream services.

- `booking-service` (`:8082`, PostgreSQL + Redis)
  - Seat catalog read APIs and booking write APIs.
  - Handles locking seats, confirming bookings, and releasing expired/abandoned locks.
  - Consumes payment success events from RabbitMQ.

- `payment-service` (`:8083`, PostgreSQL)
  - Creates payment order records.
  - Verifies payment and publishes payment success events.

- `notification_service` (`:8084`, RabbitMQ + SMTP)
  - Consumes payment success events.
  - Sends confirmation emails.

## Infrastructure Components
- PostgreSQL: persistent relational data for booking and payment services.
- MongoDB: user profile and auth data.
- Redis: distributed seat lock window for concurrency-safe booking.
- RabbitMQ: event bus for payment success fan-out.

## Gateway Routes
Configured in `api-gateway/src/main/resources/application.properties`.

- User/auth:
  - `/auth/**`, `/users/**` -> `user-service`
  - `/login`, `/register` -> rewritten to `/auth/login`, `/auth/register`
- Booking:
  - `/bookings/**`, `/catalog/**`, `/admin/**` -> `booking-service`
- Payment:
  - `/payments/**` -> `payment-service`
- Notification/testing:
  - `/notifications/**`, `/test-email` -> `notification_service`

## End-to-End Runtime Flows

## 1) Authentication Flow
1. Client calls `POST /auth/register` or `POST /auth/login` on gateway.
2. Gateway routes to `user-service`.
3. `user-service` validates payload/credentials and returns JWT.
4. Client uses `Authorization: Bearer <token>` for protected APIs.

## 2) Seat Lock + Checkout Flow
1. Client calls `POST /bookings/lock-seats` with JWT.
2. Gateway routes to `booking-service`.
3. Booking service:
   - Validates seat availability in DB.
   - Creates Redis locks for requested seats.
   - Persists `LOCKED` booking rows with expiry window.
4. Returns generated `bookingIds` to client.

## 3) Payment Verification Flow
1. Client calls `POST /payments/create` with `bookingIds` + amount and JWT.
2. `payment-service` persists payment order record with user identity.
3. Client submits payment result to `POST /payments/verify`.
4. `payment-service` marks payment as `SUCCESS` and publishes RabbitMQ event:
   - `bookingIds`
   - `paymentId`
   - `userEmail`
   - `status`

## 4) Post-Payment Fan-out
- `booking-service` consumer receives payment success event:
  - Confirms all matching `LOCKED` bookings.
  - Transitions seat states `LOCKED -> BOOKED`.
  - Releases Redis locks.

- `notification_service` consumer receives same event:
  - Uses `userEmail` from event payload.
  - Sends booking confirmation email with ticket/payment details.

## Security Model
- JWT generated by `user-service`, validated by gateway and downstream protected services.
- Public endpoints:
  - Auth and health/actuator routes as configured per service.
- Protected endpoints:
  - Booking mutations and payment APIs require valid JWT.

## Docker Networking Model
- All services run on same Docker bridge network (`npcine_net`).
- Inter-service calls use container DNS names (for example, `user-service`, `rabbitmq_broker`), not `localhost`.

## Operational Notes
- Keep JWT secret aligned across gateway + booking + payment + user services.
- Keep RabbitMQ host configured as container host in Docker deployments.
- Rebuild/restart services after config changes:
  - `docker compose up -d --build`
